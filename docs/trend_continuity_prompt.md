## Part D：反思闭环 [SHOULD]（核心——反思必须驱动推演进化）

这是整个系统最核心的竞争力——让推演引擎具备自我纠错和持续进化的能力。

### D1：反思引擎
每天生成推演后，对比前一天的推演结果和实际走势：

**推演正确时：**
- 记录：哪个场景预测正确（A/B/C）
- 分析：正确的原因是什么？
  - 趋势延续性强？（连续多日同一方向）
  - 关键位支撑有效？（前低没破/前高突破了）
  - 量能确认充分？
- 提取：可复用的"正确模式"，标记为高置信度规则

**推演错误时（更重要）：**
- 记录：预测了哪个场景，实际发生了什么
- 定位根因：
  - 状态误判？（实际趋势比判定的更强/更弱）
  - 关键位判断失误？（前高/前低位置不准）
  - 突发因素？（放量暴跌/暴涨=不可预测）
  - 参数问题？（MA20窗口、量能阈值等）
- 分类：可优化的问题 vs 不可控的随机事件
- 输出：具体的改进建议

### D2：反思数据持久化
推演日志格式（`dashboard/data/reflection_log.json`）：
```json
{
  "date": "2026-05-29",
  "projections": [
    {
      "code": "881121", "name": "半导体",
      "current_state": 3,
      "predicted": {"scene": "A", "next_state": 3, "confidence": 0.7},
      "actual": {"next_state": 4},
      "correct": false,
      "reflection": {
        "error_type": "underestimated",
        "root_cause": "结构+持续性双强但量能弱→实际放量确认了",
        "lesson": "状态3+结构持续性双OK时，即使量能弱也应上调至B场景(状态4)"
      }
    }
  ],
  "summary": {
    "total": 90, "correct": 62, "accuracy": 68.9,
    "top_errors": ["量能判断过于保守(8次)", "前高识别滞后(5次)"]
  }
}
```

### D3：改进反馈（核心——反思必须驱动推演进化）

**被动记录毫无意义。反思的唯一价值是提升下一次推演的准确率。**

具体机制：

**1. 规则权重自动调整**
- 推演引擎维护一个可调参数表（JSON文件）：`dashboard/data/projection_weights.json`
- 格式：
```json
{
  "rules": [
    {"condition": "state==3 AND struct_ok AND persist_ok AND NOT volume_ok",
     "场景A权重": 0.3, "场景B权重": 0.5, "场景C权重": 0.2,  // ← 初始权重
     "recent_accuracy": [true, false, true, true],  // 最近4次验证
     "trend": "underestimated"},  // 系统性低估
    {"condition": "state==4 AND above_ma20 AND consecutive_drop",
     "场景A权重": 0.6, "场景B权重": 0.3, "场景C权重": 0.1}
  ]
}
```
- 每次验证后：如果某规则连续N次预测错误方向（如总是低估），自动上调对应场景权重
- 例如：`state3+struct+persist` 模式连续3次实际走了状态4而非3 → 场景B权重从0.5升到0.65

**2. 规则发现**
- 反思发现新的高置信度规则时，自动添加到权重表
- 例如："状态3+MA20偏离>3%+连阳>=5天 → 80%概率次日进入状态4" → 成为新规则
- 同时记录规则的置信度和样本量

**3. 衰减机制**
- 旧规则的权重随时间衰减（市场环境在变）
- 超过30天未触发的规则自动降权50%

**4. Dashboard展示**
- 不是展示"记录了100条反思"——那毫无意义
- 展示："推演引擎已学习12条规则，近10天准确率从62%提升到74%"
- 展示最近调整的3个参数及其效果

### D4：健康度仪表
在Dashboard新增"🔄 系统健康度"面板：
- 推演准确率趋势（折线图）
- 最近改进项列表
- 参数调整建议（基于反思数据自动生成）

## 实现注意事项（关键——避免踩坑）

### ⚠️ 1. ScenarioEngine 需要重构以接受外部权重

当前 `ScenarioEngine.generate(ts)` 是纯静态方法，场景和概率完全硬编码。反思闭环要"反馈权重"但没接口。

**必须改造**：让 ScenarioEngine 接受可选的 `weights` 参数：
```python
ScenarioEngine.generate(ts, weights=projection_weights)
```
优先使用 weights 中的自定义概率，没有则回退到硬编码默认值。

### ⚠️ 2. 预计算历史状态缓存（性能关键）

822天×90板块 = 73,890次状态机 classify() 调用。每次截断数据到当天再跑，预估20-30分钟。如果推演+验证+反思多轮迭代，重复计算不可接受。

**第一步先跑预计算脚本**：生成 `dashboard/data/history_states.json`
```json
{
  "2023-01-03": {"881121": 1, "881273": 2, ...},
  "2023-01-04": {"881121": 1, "881273": 2, ...},
  ...
}
```
后续推演验证直接从缓存读状态，毫秒级。这个文件约30MB，不嵌入HTML，放 data 目录即可。

### ⚠️ 3. Dashboard 只嵌入汇总数据

73,890条推演结果不能全嵌入HTML（会撑爆）。策略：
- HTML内嵌：准确率趋势（每日一个点，822个点约15KB）、错误类型分布、Top规则、准确率变化
- 详情按需：点击某个日期时，从 `trend_snapshot_{date}.json` 加载（这些文件已存在）

### ⚠️ 4. 日期导航扩展到30天

当前日期栏只显示10天。有了822天数据后，Dashboard 日期栏改为：
- 默认显示最近30天
- 提供"查看更多"展开全部历史
- 每周一显示月份标签（如"5月"）作为分隔
